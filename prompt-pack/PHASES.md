# UDD Default Phase Map

This file defines the default phase ladder for building UDD to hardened production.

Codex may internally decompose a phase into parallel workstreams.
Codex may not change the phase intent unless explicitly justified against canonical docs.

## Phase 1 — Hosted web shell and semantic product frame

Goal:

- establish the hosted web shell
- establish semantic DOM structure
- establish route and frame composition
- establish primary product surfaces as usable shells
- establish status-bearing layout zones for runtime, preview, AI, providers, and iOS readiness

Internal workstream examples Codex may include inside the phase packet:

- shell layout and navigation
- route skeletons and project frame
- semantic status and header system
- shared empty, loading, degraded state primitives

Phase gate:

- hosted web shell is real, semantic, navigable, and not template-like
- major product surfaces exist as structurally correct shells
- no generic SaaS drift
- no material-system violations in shell composition

## Phase 2 — UI material system foundation

Goal:

- establish the canonical material primitives and fallback path

Internal workstream examples Codex may include inside the phase packet:

- GlassCardShell
- SpectralMaterialPlane
- RefractiveToggle
- LightFieldController
- ForegroundContent integration
- quality-tier scaffolding
- reduced-motion and fallback behavior

Phase gate:

- core material primitives exist
- DOM-first plus WebGL-material boundary is preserved
- fallback tier exists
- reduced-motion path exists
- shader system is real, not faked with flat CSS as the primary path

## Phase 3 — Flagship project view

Goal:

- make the project view the flagship proving ground for the product and material system

Internal workstream examples Codex may include inside the phase packet:

- project view center composition
- AI panel integration
- runtime and preview state framing
- iOS readiness and compatibility indicators
- logs and activity strip
- provider status integration
- limited, beta, implementation-gap state treatment

Phase gate:

- project view is the strongest, most complete screen in the product
- runtime, preview, AI, provider, and iOS states are clearly visible
- material system is integrated, not pasted on
- states are production-complete within the project view scope

## Phase 4 — Operational and configuration surfaces

Goal:

- complete the supporting operational product surfaces around the project view

Internal workstream examples Codex may include inside the phase packet:

- runtime screen
- preview screen
- providers and secrets screen
- settings
- web + iOS continuity screen
- deploy and export as secondary surface

Phase gate:

- operational surfaces are usable and coherent
- secrets and provider flows are readable and trustworthy
- preview and runtime surfaces feel central to the hosted product
- settings and deploy/export do not overpower the core workflow

## Phase 5 — Project creation, stack-aware flow, and contract discipline

Goal:

- make project creation and stack-aware flows real enough to support the documented product loop
- strengthen web and iOS compatibility and contract discipline

Internal workstream examples Codex may include inside the phase packet:

- new project flow
- bundled template path
- stack choice and stack-aware UI
- project initialization states
- web+iOS contract-facing checks or UI hooks
- parity and conformance-oriented UI support

Phase gate:

- project creation path is coherent
- stack-aware UX is visible and structured
- contract and parity obligations are surfaced clearly
- no fake completeness where implementation gaps remain

## Phase 6 — Hardening to production

Goal:

- harden the implemented surfaces and systems to production expectations inside the current architecture

Internal workstream examples Codex may include inside the phase packet:

- performance hardening
- quality-tier tuning
- reduced-motion polish
- fallback polish
- state completeness audits
- accessibility hardening
- visual and system consistency hardening
- regression-sensitive cleanup within existing meaning

Phase gate:

- product surfaces built so far are hardened
- state coverage is complete
- accessibility is not an afterthought
- quality tiers are credible
- no remaining known UI-level shortcuts inside the built scope
